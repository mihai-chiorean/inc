# Schemas — manifest, lockfile, overlay

This document specifies the three file formats used by `/staff`. Source of truth is this doc; example files live in `../examples/`.

## Vocabulary

- **HR repo** — this repo (`claude-agents`). Canonical source of agent definitions. Per-project terminology only; not load-bearing.
- **Stable ID** — short hyphen-separated identifier for an agent (e.g., `go-engineer`). Survives file renames and category moves. The lockfile and manifest both key on stable ID.
- **Manifest** — `agent.manifest.yaml` at HR repo root. Lists every agent with stable ID, current file path, content hashes, tags, project hints, declared conflicts. Regenerated by `scripts/generate-manifest.py` whenever agent files change.
- **Lockfile** — `.claude/staff/lock.yaml` in a project. Records which agents are staffed at which HR commit, with content hashes pinned at apply time.
- **Overlay** — `.claude/staff/overlays/<id>.md` in a project. Append-only project-specific context for one agent. Owned by the project, never synced from HR.
- **Merged agent file** — `.claude/agents/<id>.md` in a project. Generated artifact: HR base content + overlay content stitched together with marker comments. Claude Code loads this file. Do not hand-edit.

## Directory layout in a staffed project

```
<project-root>/
  .claude/
    agents/
      go-engineer.md           # GENERATED — Claude Code loads this. Do not edit.
      swift-backend.md         # GENERATED.
    staff/
      lock.yaml                # Lockfile (committed if .claude/agents/ is committed)
      config.yaml              # Project-local /staff config: HR repo path, etc.
      overlays/
        swift-backend.md       # Overlay source — user edits this.
        # No overlay file means no project-specific context for that agent.
```

Why split `.claude/staff/` from `.claude/agents/`: Claude Code walks `.claude/agents/` for all `.md` files and loads each as an agent. Anything we don't want loaded — overlay sources, lockfile, config — must live elsewhere. `.claude/staff/` is the staging area; `.claude/agents/` only contains what Claude actually loads.

## Schema 1 — `agent.manifest.yaml` (HR repo root)

```yaml
schema_version: 1
generated_at: 2026-05-07T18:00:00Z
generated_by: scripts/generate-manifest.py
source_repo: claude-agents

agents:
  go-engineer:                                       # stable ID — the dictionary key
    file: engineering/go-engineer.md                 # current path within repo
    category: engineering                            # source folder
    description: |
      Use this agent when writing or modifying Go code, building Cobra-based CLIs...
    description_hash: sha256:a7f9c2...               # of the description string above
    body_hash: sha256:b3e8d1...                      # of the system prompt body (everything after frontmatter)
    tags:
      - go
      - cli
      - cobra
      - cross-compile
    project_hints:
      files:                                         # filenames whose presence suggests this agent
        - go.mod
        - go.sum
      regex:                                         # regex patterns matched against CLAUDE.md / README
        - "(?i)\\bgolang?\\b"
        - "(?i)\\bcobra\\b"
    conflicts: []                                    # list of stable IDs declared to overlap (e.g., ["backend-architect"])
    introduced: 2026-04-15                           # YYYY-MM-DD when first added
    aliases: []                                      # past stable IDs, for rename history
```

### Field semantics

| Field | Required | Generated by script | Hand-curated | Notes |
|---|---|---|---|---|
| `file` | yes | yes | no | Updated by the generator when files move |
| `category` | yes | yes | no | Inferred from folder |
| `description` | yes | yes | no | Pulled verbatim from frontmatter `description:` |
| `description_hash` | yes | yes | no | Sync-detection signal |
| `body_hash` | yes | yes | no | Sync-detection signal |
| `tags` | yes | seeded from name + category | yes (refine) | Matchable keywords, lowercase-hyphen. `/staff suggest` matches against these |
| `project_hints.files` | no | no | yes | Filenames in a project root that strongly suggest this agent |
| `project_hints.regex` | no | no | yes | Regex patterns evaluated against CLAUDE.md and README in the project. Case-insensitive recommended |
| `conflicts` | no | no | yes | Other stable IDs known to overlap. Used by eval framework's conflict detector and as a hint to /staff suggest |
| `introduced` | yes | once (initial generation) | no | Git-archaeology source for "is this a new agent" |
| `aliases` | no | no | yes | Filled when an agent is renamed; preserves continuity for old lockfiles |

### Renames

When a file is renamed (e.g., `engineering/embedded-device.md` → `embedded/jetson-agent.md`):

1. Stable ID stays the same: `embedded-device`.
2. `file:` field updates to the new path automatically on next `generate-manifest.py` run.
3. If the stable ID itself changes (rare — only when the agent's purpose has fundamentally shifted): **operator action required.** Before regenerating the manifest, hand-edit the entry to add the old ID to `aliases:` of the new entry. The generator does NOT auto-detect ID renames; it has no way to know that `agent-A` becoming `agent-B` is a rename vs. `agent-A` being deleted and `agent-B` being added. After the alias is in place, `/staff sync` will resolve old lockfile entries to the new ID.

`/staff sync` resolves agents in a lockfile by stable ID. If not found at top level, it scans `aliases:` of every manifest entry.

## Schema 2 — `.claude/staff/lock.yaml` (per-project)

> **`lock.yaml` is generated by `/staff apply`. Do not hand-edit.** It is committed to the project repo so the staffing state is reviewable in PRs and reproducible across machines, but it is regenerated on every apply — manual edits will be overwritten without warning. The user-owned source of truth is `config.yaml` and the overlay sources under `overlays/`.

```yaml
schema_version: 1
hr_repo: file:///home/mihai/workspace/claude-agents
hr_commit_pinned: 39ff68d1234567890abcdef
generated_at: 2026-05-07T18:23:00Z
generated_by: /staff apply

staffed:
  go-engineer:
    pinned_at: 39ff68d1234567890abcdef0123456789abcdef0      # full 40-char HR commit when last apply'd
    file: engineering/go-engineer.md                          # path in HR at pin time
    description_hash_at_pin: sha256:a7f9c2...                 # description hash from manifest at pin time
    body_hash_at_pin: sha256:b3e8d1...                        # body hash from manifest at pin time
    generated_hash_at_apply: sha256:7f8e9d...                 # sha256 of generated .claude/agents/<id>.md
    overlay: false
  swift-backend:
    pinned_at: b651e49abcdef0123456789abcdef0123456789ab
    file: engineering/swift-backend.md
    description_hash_at_pin: sha256:e1f2a3...
    body_hash_at_pin: sha256:c1d2e3...
    generated_hash_at_apply: sha256:abcdef...
    overlay: true
    overlay_path: .claude/staff/overlays/swift-backend.md
    overlay_hash_at_apply: sha256:0a1b2c...                   # sha256 of overlay body at apply time
    overlay_last_reviewed: 2026-05-07                         # mirrored from overlay frontmatter at apply time
```

### Field semantics

| Field | Notes |
|---|---|
| `hr_repo` | Path to canonical source. Local file URL for now; could be a git URL later |
| `hr_commit_pinned` | Full 40-char SHA. The most recent HR commit `/staff apply` was run against. NOT necessarily the commit each individual agent was pinned at — see per-agent `pinned_at` |
| `staffed.<id>.pinned_at` | Full 40-char SHA. The HR commit that `body_hash_at_pin` and `description_hash_at_pin` were sampled from. May lag `hr_commit_pinned` if the agent's content didn't change between commits |
| `staffed.<id>.description_hash_at_pin` | Description hash from HR manifest at pin time. Routing-critical — `/staff status` uses this to detect description-only changes that the body hash wouldn't surface |
| `staffed.<id>.body_hash_at_pin` | Body hash from HR manifest at pin time. Used to detect HR-side body changes since the last apply |
| `staffed.<id>.generated_hash_at_apply` | sha256 of the generated `.claude/agents/<id>.md` written by `/staff apply`. Lets `/staff status` detect manual tampering with the merged file by comparing against the file currently on disk. **This is what makes manual-edit detection unambiguous** — no need to reconstruct from HR + overlay |
| `staffed.<id>.overlay` | Boolean. True if `.claude/staff/overlays/<id>.md` exists at apply time |
| `staffed.<id>.overlay_path` | Path to overlay source, when `overlay: true` |
| `staffed.<id>.overlay_hash_at_apply` | sha256 of the overlay body content at apply time. `/staff status` uses this to flag "overlay edited but not re-applied" |
| `staffed.<id>.overlay_last_reviewed` | Snapshotted from overlay frontmatter at apply time. `/staff status` warns if the live overlay frontmatter is older than `stale_overlay_days` (default 90) |

### Why per-agent `pinned_at` AND top-level `hr_commit_pinned`

Top-level says "the last time /staff apply was run, HR was at this commit." Per-agent `pinned_at` says "this specific agent's body matched HR's body at this commit." When you re-apply, only the changed agents need their pin bumped. The two together let `/staff status` report: "HR has advanced 14 commits since last apply, but only 3 of your staffed agents had any changes."

## Schema 3 — overlay (`.claude/staff/overlays/<id>.md`)

```markdown
---
agent_id: swift-backend
last_reviewed: 2026-05-07
notes: "Wendy cloud-services context"
---

## Project Context

This codebase uses mTLS via ALB header forwarding (X-Client-Cert).
gRPC services use Hummingbird 2 with bidirectional streaming for the
device telemetry RPC. Postgres access goes through postgres-nio with
the pool config in Sources/CloudServices/Database/PoolConfig.swift.

Never use Foundation in the auth path — see ADR-014.
```

### Frontmatter fields

| Field | Required | Notes |
|---|---|---|
| `agent_id` | yes | Stable ID this overlay applies to. Must exist in HR manifest |
| `last_reviewed` | yes | YYYY-MM-DD. `/staff status` warns if this is > 90 days old |
| `notes` | no | Free-form short label, e.g., the project name or context |

### Body

Append-only context. Plain markdown. The body is appended verbatim to the merged agent file body. Recommend leading with `## Project Context` (or similar H2) so it's clearly delimited from the HR base content.

## Merge format — `.claude/agents/<id>.md` (generated)

```markdown
---
name: swift-backend
description: ...
model: sonnet
---

<HR base body content here>

<!-- BEGIN STAFF OVERLAY -->
<!-- agent_id: swift-backend -->
<!-- overlay_source: .claude/staff/overlays/swift-backend.md -->
<!-- overlay_last_reviewed: 2026-05-07 -->
<!-- DO NOT EDIT THIS FILE — edit the overlay source and run /staff apply -->

<overlay body content here>

<!-- END STAFF OVERLAY -->
```

Marker comments (`BEGIN/END STAFF OVERLAY`) let `/staff apply` detect and replace the overlay block on regeneration without having to re-fetch HR base. `/staff status` uses these markers to detect manual edits to merged files (anything outside the markers that doesn't match HR HEAD is a manual edit warning).

If no overlay exists for a staffed agent, the merged file is just the HR base verbatim with no marker block.

## Hashing

All hashes are `sha256:` prefixed hex digests over UTF-8 bytes of a **canonicalized** input string.

**Canonicalization:** strip leading and trailing whitespace (`s.strip()`). No internal normalization — a difference of a single internal space matters.

This applies uniformly to: `description_hash`, `body_hash`, `generated_hash_at_apply`, `overlay_hash_at_apply`, and any future hash field.

Reference implementation: `canonicalize_for_hash()` and `sha256()` in `scripts/generate-manifest.py`. `/staff` operations MUST share this canonicalization or hashes will drift.

### Per-field input

| Hash field | Input |
|---|---|
| `description_hash` | The `description:` value extracted from agent frontmatter |
| `body_hash` | Everything in the agent file after the closing frontmatter `---` line |
| `generated_hash_at_apply` | The full content of `.claude/agents/<id>.md` as written by `/staff apply` |
| `overlay_hash_at_apply` | Everything in the overlay file after the closing frontmatter `---` line |

## Stable ID rules

- Lowercase ASCII letters, digits, hyphens. No spaces, no slashes, no dots.
- Match the agent's frontmatter `name:` field at creation time.
- Never reused across agents. If an agent is retired, its ID is permanently dead — do not reassign.
- Renames: prefer keeping the ID and updating only the file path. Only change the ID if the agent's purpose has fundamentally shifted, in which case the old ID goes into `aliases:`.

## What this schema does NOT do (v1)

- **No version pinning per agent against HR commits other than apply-time.** You don't say "always pin go-engineer to v1.2." You staff at HR's HEAD at apply time and explicitly re-apply when you want updates.
- **No semver.** Agents don't have versions; only the HR commit hashes pin them.
- **No three-way merge.** `/staff sync` regenerates the merged file from HR HEAD + the unchanged overlay. Manual edits to the merged file are detected as drift, not preserved.
- **No multi-source HR.** One HR repo per project, pinned in `config.yaml`.
- **No capability schema.** `tags` and `project_hints` are the only structured matching fields. The free-form `description` is what actually convinces Claude to route to an agent — schema doesn't try to second-guess it.
- **No automatic stable-ID rename.** If an agent's stable ID changes, an operator must hand-add the old ID to `aliases:` before regenerating. The generator cannot distinguish a rename from a delete+add.
- **Auto-derived `tags` are weak.** They're seeded from name parts and the category folder. `/staff suggest` should not rely on them alone — `project_hints` is the load-bearing match field, and tags are a fallback that benefit from hand-curation. MIT-281 will populate hints for the most-used agents.
